Requirements

Functional Requirements

The following are the functional requirements for the system:


  • FR-1. The application shall allow caregivers and patients to register an account.
  • FR-2. The application shall provide a user interface to allow its user to access their metrics and update their personal information.
  • FR-3. The application shall enable notifications to be sent to the requested parties.
  • FR-4. The application shall interface with various technologies such as wearable technologies, mobile phones, and desktop computers.
  • FR-5. The application shall allow direct communication between the doctor and caregiver.
  • FR-6 The application shall warn the caregiver and doctor if the patient's blood oxygen decreases rapidly.
  • FR-7 The application shall warn the caregiver and doctor if there are any medical anomalies.
  • FR-8. The application shall support patients' and caregivers' ability to create appointment reminders and alerts for a wide range of medical issues and tasks.
  • FR-9. The application shall support external fingerprint scanning for extra security.
  • FR-10. The application shall have a backup way to sign in and out in case of a fingerprint scanner error.
  • FR-11. The application shall exist between different mobile operating systems such as iOS and Android.
  • FR-12. The application shall provide geolocation to help find a patient in case of an emergency.
  • FR-13. The application shall allow bidirectional synchronous communication between the user and the data source in real-time.

Non-Functional Requirements

These Non-Functional requirements impose constraints under which the application we are developing has to function. The following are the requirements that we are explicitly defining and hold for each customer instance.

Architecture:

  • NFR-1. Allow for disparate data sources to be ingested in a multi-tenant and performant manner. (Apache Kafka or Kinesis streams)
  • NFR-2. Multiple location cloud production environments.
  • NFR-3. Containerized services for quick swapping/ability for hybrid strategy in the future.
  • NFR-4. Web and mobile endpoints that help collect and analyze data, multi-role users.

Security:

  • NFR-5. The application must comply with HIPAA regulations.
  • NFR-6. Only run on specific versions of devices for security reasons.
  • NFR-7. Data storage must conform to HIPAA and FEDRAMP requirements.

Safety:

  • NFR-8. Data from the database will never be transmitted to unauthenticated sources.
  • NFR-9. The application will never disclose patient information to users that are not authorized to see it.

Availability:

  • NFR-10. The application should be available 99.99% of the time.

Performance:

  • NFR-11. The application shall take no more than 2 seconds to access.
  • NFR-12. The application's fingerprint scanning shall recognize registered users 95% of the time.
  • NFR-13. The application shall take the appropriate measures to ensure data servers are secured.
  • NFR-14. Data backups should be created daily (minimum).
  • NFR-15. Data backups should be securely stored separately.
  • NFR-16. A private REST API should be available for third-party, desktop, or mobile application integration.
  • NFR-17. The access to all data should take a least privilege approach.
  • NFR-18. Data streaming should be processed near real-time.
  • NFR-19. The application shall have the ability to change out algorithms for smart altering.
